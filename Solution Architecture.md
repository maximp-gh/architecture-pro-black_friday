 ### Коллекция orders

#### Примерная схема:

```json
{
    "_id": {"type": "ObjectId", "descripttion": "Уникальный идентификатор заказа"},
    "customer_id": {"type": "ObjectId", "descripttion": "Идентификатор владельца или сессии заказа"},
    "created_at": {"type": "Date", "descripttion": "дата создания"},
    "product_list": { "type": "array", "descripttion": "список product_id в заказе c name, price , quantity"},
    "order_status": {"type": "string", "descripttion": "статус заказа"},
    "order_totalprice": {"type": "Decimal", "descripttion": "общая стоимость заказа"},
    "order_geozone": {"type": "string", "descripttion": "Геозона заказа"},
}
```

Элементы product_list такие как name, price, quantity хранятся внутри документа, чтобы гарантировать правильность истории заказа даже если товар в каталоге позже будет удален или переименован.

 #### Индексы.

{ "customer_id": 1, "created_at": -1 } — для выборки истории заказов пользователя.

 #### Шардинг
 
 Планируется шардировать коллекцию по ключу {"customer_id": hash}. 

### Коллекция products:

#### Примерная схема:

```json
"_id": {"type": "ObjectId", "descripttion": "Уникальный идентификатор продукта"},
"name": {"type": "string", "descripttion": "название продукта"},
"category": {"type": "string", "descripttion": "Категория, например Электроника"},
"price": { "type": "Decimal", "descripttion": "цена"},
"amount_per_geozone": {"type": "array", "descripttion": "список кол-ва по геозонам"},
"details": {"type": "Object", "descripttion": "детали продукта - цвет, размер, и т.д."},
```

#### Индексы:

{ "category": 1 }
{ "price": 1 }
{ "amount_per_geozone": 1 }

#### Шардинг
Шардирование по категории - самое простое. Но name имеет явно большую кардинальность.

    "product_list": { "type": "array", "descripttion": "список product_id в корзине c ценой и кол-вом"},

    "expires_at": {"type": "Date", "descripttion": "время для автоматического удаления"},

{"customer_id": "hashed", status: 1}

1. Анализ

Судя по тому, что перегрузка произошла из-за определенной категории товаров и только на одном шарде, можно сделать вывод, что шардирование происходило по категориям (products.category) товаров (диапазонное шардирование).
Для более равномерного распределения стоит рассмотреть варианты:
  - увеличение кол-ва категорий продуктов для повышения кардинальности. Наприме категорию "Электроника" разделить на Телефоны, Планшеты, Настольные Компьютеры, Ноутбуки и т.д., но риск перегрузки более детальной категории все равно сохранится.
- изменение ключа шардирования products для повышения кардинальности. Например name. reshardCollection(). Возможно использование compound key. Это также может позволить использовать большее кол-во шардов (addShard()).

- Как временная мера может подойти вертикальное масштабирование горячего шарда или принудительная перебалансировка (может быть невозможна - зависит от ключа шардирования).

2. Мониторинг

Предлагаются следующие метрики для постоянного мониторинга и возможности автоматического оповещения:

| **Метрика** | **Возможный порог оповещения** | **Действие** |
|:-:          |:-----:|:-:|
| CPU на шарде | 80%  | Вертикальное масштабирование |
|Disk Space    | 70%  | Вертикальное масштабирование |
| Распределение документов по шардам | > 20% от среднего  | Изменение ключа шардирования  - например не по категориям товаров , а по product _id. Динамическое шардирование |
| Среднее время выполнения запросов на шарде | > 20% от среднего | Уточняющий анализ |
| Кол-во медленных запросов |> 100мс | Уточняющий анализ |
| Кол-во запросов на шард | > 20% от среднего | Скорее всего понадобится ребалансировка, решардинг или добавление шардов. Динамическое шардирование.|


Нужно проводить регулярный анализ паттернов доступа и аудит распределения данных. Указанные метрики помогут в этих задачах.

- Проверять статус шардирования (sh.status)
- Проверять статус балансировщика (sh.getBalancerState()) и хронологию динамической перебалансировки. Она не должна по возможности совпадать с пиковыми часами. Настроить окна для перебалансировки.
- Отслеживать метрики и планировать действия, не дожидаясь порога оповещения.  Например, при постоянно растущем допороговом значении.

3. Автоматизация


 - Добавление шардов (addShard), при условии, что коллекция имеет хороший ключ с высокой кардинальностью. Кол-во шардов, превышающее кардинальность, не имеет смысла.

 - автоматическое вертикальное масштабирование в облаке на время повышения активности

 - принудительная перебалансировка (moveChunk)

Допустимая задержа репликации: минуты. Попадание нового заказа в историю не выглядит критичной ко времени.

Статус заказа критичен для пользователя и должен читаться с мастера.

| **Операция чтения** | **PRIMARY/SECONDARY** |
|:-:                  |:-----:|
| Кол-во продукта в геозоне | PRIMARY |
| Описание продукта | SECONDARY | 
| Фильтрация по диапазону цен | PRIMARY |
| Поиск по категориям | SECONDARY | 

Кол-во продукта и цены должны быть актуальны, иначе заказ потеряет смысл для пользователя - PRIMARY.

Допустимая задержа репликации: секунды. Изменение в описании продукта или его категории не очень часты и критичны.
